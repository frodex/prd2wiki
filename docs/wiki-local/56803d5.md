## Phase 3a: Tree Infrastructure (~12h)

### 3a.1: Tree scanner (`internal/tree/`)

Scan `tree/` recursively. Find `.uuid` (projects) and `.link` (pages). Build in-memory map. Rescan on SIGHUP.

### 3a.2: URL router rewrite

**New:** `GET /{tree-path}` → walk tree → .link → UUID → git → render.

**Reserved prefixes — check FIRST, before tree router:**
- `/api/*` — REST API
- `/static/*` — CSS/JS/assets
- `/blobs/*` — content-addressed binary store
- `/admin/*` — export/import/verify dashboards
- `/projects/*` — legacy URL redirects (301 to tree paths)
- `/health`, `/debug/*` — ops endpoints

**Path security:**
- Normalize URL path BEFORE filesystem walk
- **Reject** any segment containing `..`, empty segments, or encoded traversal (`%2e%2e`, `%2f`)
- Resolve `tree.dir` to absolute path once at startup; refuse any resolved path outside it
- **Symlinks under `tree/`:** ignore or reject in production (don't follow)
- Validate `.link` / `.uuid` / `.301` / `.302` filenames — no user-controlled path segments reach the filesystem without sanitization

**Redirect handling:** At each path segment, check for `.301`/`.302`. HTTP redirect with remaining path appended.

**Legacy URLs:** `GET /projects/{project}/pages/{id}` → 301 to tree path.

### 3a.3: Project discovery

Remove `projects:` from config. Scan tree for `.uuid` at runtime. No restart.

### 3a.4: Page CRUD via tree

**Create:** Generate UUID → write `pages/{uuid}.md` to git → create `{slug}.link` (line 2 empty) → `syncToLibrarian` → write .link line 2.

**Edit:** Write to git → `syncToLibrarian` → update .link line 2.

**Delete:** `rm {slug}.link` → async `memory_delete` by `page_uuid` on librarian. If librarian down, compactor cleans orphans on reconcile. Pages removed from search immediately (tree scan excludes them).

### 3a.5: API migration

**New endpoints:**
```
GET    /api/tree                           → list projects + pages
GET    /api/tree/{path...}                 → page or directory listing
POST   /api/tree/{project-path}/pages      → create page
PUT    /api/tree/{path...}/{slug}          → update page
DELETE /api/tree/{path...}/{slug}          → delete page
GET    /api/tree/{path...}/{slug}/history  → version history
POST   /api/tree/{path...}/{slug}/status   → change status
```

**Legacy API:** Old `/api/projects/{project}/...` → **308 Permanent Redirect** (preserves HTTP method). Parallel routes during transition with `Deprecation` header. Remove after no traffic.

**Auth on mutating endpoints:** `POST`, `PUT`, `DELETE`, blob upload, and all `/admin/*` endpoints MUST require API key scope. Anonymous read may be OK for public wiki — explicit product decision, not accidental default. Audit every route after migration.

### 3a.6: Blob store

`POST /api/blobs` → SHA-256 → `data/blobs/{hash[0:2]}/{hash}` → return `/blobs/{hash}`.
`GET /blobs/{hash}` → serve with content-type detection.

**Security:**
- **Max upload size:** enforce per-request body limit (e.g., 50MB). Reject oversized.
- **Hash validation:** strict hex, exact length. No user input reaches filesystem path without validation. Prevents `../` in hash param.
- **Content-type sniffing:** Set `X-Content-Type-Options: nosniff`. Serve unknown types as `application/octet-stream`. Never serve user uploads as `text/html` (XSS vector).
- **Deduplication:** Same hash = same file. No re-upload needed.

### 3a.7: Wire libclient + syncToLibrarian

Create `internal/libclient/client.go` (remote librarian client, socket + ticket auth). Wire into edit flow.

**Codebase note:** Today's `SubmitRequest` has `Project`, `Branch`, `Frontmatter`, `Body`, `Intent` — NO `PageUUID`, `ProjectUUID`, or `syncToLibrarian`. Pre-flight Item 1 adds the UUID fields (empty until tree exists). This step wires them to the libclient. The edit path doesn't work end-to-end until both land.

**Socket permissions:** Only the wiki service user and ops should have access to `/var/run/pippi-librarian.sock`. Not world-readable. Document failure mode if misconfigured (silent sync failure, retry queue fills).

**Ticket scope:** `memory_store` and `memory_delete` scopes only. A compromised wiki key must not be able to call unrelated librarian tools.

## Phase 3c: Management Tools (~6h)

### 3c.1: Move page
`mv .link` + `.302`. CLI + MCP `wiki_move` + UI.

### 3c.2: Rename slug
`mv .link` + `.301`. CLI + MCP `wiki_rename` + UI.

### 3c.3: Move project
`mv` directory + `.302`.

### 3c.4: Two MCP schema tracks

| Track | Codebase | Tools | Addressing |
|-------|----------|-------|-----------|
| **Librarian `memory_*`** | pippi-librarian | store, search, get, delete | `page_uuid`, `record_id`, namespace |
| **Wiki `wiki_*`** | prd2wiki | search, read, propose, challenge, ingest, lint, status, **move**, **rename** | tree paths |

Separate codebases, separate schemas. Integration tests must refuse cross-wiring (wrong tool name → wrong addressing).

**Updated wiki tools:** wiki_search (add `path` filter), wiki_read (accept tree path or UUID), wiki_propose (accept tree path, create .link), wiki_move (NEW), wiki_rename (NEW), plus existing lint/challenge/ingest/status.

## Security Checklist (must-do before production)

These are non-negotiable. The builder must address each one:

- [ ] **Socket permissions:** `/var/run/pippi-librarian.sock` readable only by wiki service user + ops
- [ ] **Ticket scopes:** wiki key can only call `memory_store` + `memory_delete`, not all librarian tools
- [ ] **Route auth audit:** every mutating endpoint requires API key scope after migration
- [ ] **Path traversal:** reject `..`, `%2e%2e`, empty segments, symlinks in tree router
- [ ] **Blob upload:** max body size enforced, hash strictly validated, `X-Content-Type-Options: nosniff`
- [ ] **Import tarball:** sanitize paths on extract, no absolute paths, no `../`, manifest verification
- [ ] **Admin CSRF:** require API key header (not cookie auth) for `/admin/*` mutating actions
- [ ] **Export secrets:** verify export code NEVER includes service keys, `.env`, or BadgerDB auth data
- [ ] **Concurrency:** `MemoryStore.Store()` single-writer mutex — do not call concurrently for same `page_uuid`

## Build Strategy (recommended order within Phase 3a)

The current codebase has none of the tree infrastructure. This is the recommended order to avoid integration pain:

1. **Tree scanner** (3a.1) — can be built and tested standalone
2. **Reserved-prefix router** (3a.2) — wire alongside existing routes, not replacing them yet
3. **Project discovery** (3a.3) — replace config list, verify sidebar still works
4. **libclient + syncToLibrarian** (3a.7) — needs Phase 2 `Store()` returning `(string, error)` first
5. **Page CRUD** (3a.4) — needs libclient + tree scanner
6. **API migration** (3a.5) — new endpoints + 308 redirects for old ones
7. **Blob store** (3a.6) — independent, can land any time

**Feature flag for versioning (recommended):** Phase 2 new-chain-forward is a memory subsystem rewrite. Ship `generateMemoryID()` + `versionAndStore()` behind a config flag, test with synthetic UUIDs, then flip for production. Don't rewrite memory + tree router + UI simultaneously.
