# RESEARCH: The Head-Delete Gap — Resolved

**Date:** 2026-04-11
**Status:** Resolved — new-chain-forward strategy adopted
**Resolution:** [Version-Aware Memory Store](/projects/default/pages/c6525ac) (c6525ac)
**Context:** Raised by audit R4/R5. Resolved by building new chain forward instead of mutating head in place.

## The Problem

LanceDB "upsert" = `Delete` + `Add` — two operations, no transaction. If crash between them, the head row vanishes. See c6525ac for the full write strategy.

## Solution Chosen: New Chain Forward

Add new head first (always safe). Demote old head second. If crash, two "current" rows — startup picks highest version. No WAL, no quarantine, no operator intervention.

**Variant considered:** Mutate old head's ID to a temp value first (blocks lookups), copy to new row with original ID + new content, then give old row a new ID as the superseded snapshot. This effectively locks the row during the operation. Rejected in favor of simpler "add new, demote old" because the single-writer mutex already prevents concurrent access, and the append-only approach has fewer moving parts.

## Options Considered

| Option | Verdict |
|--------|---------|
| A: Ordered writes + startup repair | Close, but "promote snapshot" is complex |
| B: WAL in BadgerDB | Too much complexity for derived state |
| C: Append-only (duplicate IDs) | Risky — Lance duplicate-ID behavior unverified |
| D: Wipe and re-import | Correct but manual |
| E: Detect and flag | Too passive |
| **F: New chain forward** | **Adopted.** Add new head, demote old. Simple, safe, no delete gap. |
| G: Mutate-copy-restore (ID swap) | Sound but more complex than F for same safety guarantees under single-writer. |

See c6525ac for the full implementation design.
