## Cross-Repo Boundary

**`/srv/prd2wiki/docs/constraints-prd2wiki-pippi.md`** — binding constraint declaration for any work that crosses the prd2wiki ↔ pippi-librarian boundary (socket, tickets, `memory_*` tools, `libclient`, `syncToLibrarian`). Read this before touching Phase 2, 3a.7, or any code that opens a socket to the librarian.

Key rules (full detail in the file):
- **`page_uuid`** is stable identity. **`mem_`** is volatile (changes every edit).
- **New head first**, then demote old head. No Delete+Add on the new head.
- **`Store()` → `(string, error)`**. MCP adds `record_id`.
- **Git save never blocked** by librarian availability. Queue retry.
- **Two tool worlds:** Librarian `memory_*` vs Wiki `wiki_*` — separate schemas, never merged.
- **Known failures:** Lance upsert is not atomic, unpadded base-36 breaks lex order, export .link line 2 is volatile. All documented with audit references.

## Codebase Reality

**prd2wiki:** No tree router, no `PageUUID`, no `syncToLibrarian`, no `libclient`. Phase 3a is a large refactor.

**pippi-librarian:** `Store()` uses namespace+key, updates in place, returns error only. Phase 2 is a memory subsystem rewrite.

## Build Strategy

1. **Start now:** Items 1-5 (prd2wiki) + items 6-7 (pippi-librarian) in parallel
2. **Entity merge first** (item 6) before Phase 1 schema work
3. **Phase 2 behind feature flag** — test with synthetic UUIDs before production
4. **Pre-flight B: maintenance window** — wiki stopped
5. **Phase 3a.7 needs Phase 2** — `Store()` must return `(string, error)` first
6. **Don't rewrite everything at once** — versioning, then tree, then UI
7. **Read constraints file** before any cross-repo work
