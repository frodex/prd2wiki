## User's Vision

The user wants to build a system that:

1. **Ingests or creates PRDs** — can take in existing PRD documents or help author new ones
2. **Stores as a wiki** — the PRD becomes a living document, not a static file
3. **Full history and decisions** — every change, decision, and the reasoning behind it is preserved
4. **Discovery tracking** — captures research, findings, and exploration that inform the PRD
5. **Revision history** — full audit trail of what changed, when, and why
6. **Authorization audit** — who approved what, when, with what authority
7. **Reasoning encoded** — the "why" behind decisions is first-class, not an afterthought
8. **Published source of truth** — a canonical, approved version that represents current truth
9. **Ingestion pipeline** — accepts new data from various sources
10. **Conflict resolution and reasoning** — new data is evaluated, resolved, and reasoned about before promotion to truth

## Key Concepts Emerging

- **Two-tier model:** Draft/incoming vs. published/approved truth
- **Decision records:** Every change needs reasoning, not just a diff
- **Authorization gates:** Not everyone can push to truth — there's an approval workflow
- **Multi-source ingestion:** Data comes from various places (documents, conversations, research, etc.)
- **Reasoning engine:** Something that evaluates incoming data against existing truth

## Scope Clarification (from user)

This is NOT just a single PRD tool. This is **core infrastructure for agentic development**:

- **Multi-project, multi-agent** knowledge backbone
- Wiki with **branches** (like git branches for knowledge states)
- **Self-referential** — agents navigate, find what they need, understand local resources
- Agents are **encouraged to CHALLENGE** the source of truth — contradictions, errors, new research
- **Open for input**, but the source-of-truth branch is **highly curated**
- Must expose an **MCP interface** for agent interaction
- Starting from Karpathy's LLM Wiki pattern as architectural foundation

## Karpathy LLM Wiki Pattern (reference architecture)

Three layers:
1. **Raw Sources** — immutable collection of articles, papers, PDFs, documents
2. **The Wiki** — LLM-generated/maintained markdown pages (summaries, entities, concepts, syntheses)
3. **The Schema** — configuration defining wiki structure, conventions, workflows

Three operations:
1. **Ingest** — process new source, extract info, write summaries, update index, revise 10-15 related pages
2. **Query** — search wiki pages, synthesize answers with citations, optionally file answers back as pages
3. **Lint** — periodic health checks for contradictions, stale claims, orphaned pages, missing cross-refs, gaps

Supporting files: `index.md` (content catalog), `log.md` (append-only operation log)

Tools mentioned: Obsidian, Web Clipper, Marp, Dataview, qmd (local search with BM25/vector hybrid)

Key insight: "The tedious part of maintaining a knowledge base is not the reading or thinking — it's the bookkeeping." LLMs handle bookkeeping, humans curate sources and direct analysis.

## Dual-Surface Requirement

The wiki must present two seamless interfaces over the same content:

1. **Human surface** — WYSIWYG web UI where humans view, edit, and provide curation guidance. Not just reading — active participation in shaping the wiki.
2. **Machine surface** — structured, machine-readable layer for agents/code to read, write, query, and challenge content programmatically (via MCP).

Both surfaces operate on the same underlying data. Edits from either side are immediately visible to the other. Markdown is the natural storage format — it's both human-readable and machine-parseable.

This means we need:
- A web application with WYSIWYG markdown editing (not just raw markdown)
- An MCP server exposing wiki operations to agents
- A shared content layer that both surfaces read/write

## Authorization Model: Provenance-Based Trust Propagation

The authorization model is NOT simple role-based gating. It's a **source chain trust model**:

1. **Source A** provides data X. X gets vetted and accepted into truth.
2. **Agent B** builds on A's reference to create Y. Y's trust is partially inherited from A's vetted status.
3. **Agent C** can approve B's work related to A's reference — because A's data was already vetted AND has not been deprecated/contradicted elsewhere in the wiki.

Key properties:
- **Trust flows through citation chains** — vetted sources confer trust to downstream work that correctly references them
- **Deprecation breaks the chain** — if source A is later challenged or deprecated, all downstream work referencing A gets flagged for re-evaluation
- **Agents can reason about provenance** — they don't just check permissions, they evaluate whether the source chain is intact
- **Cascading invalidation** — when truth changes, the system traces what depends on it

This is a hybrid of:
- Role-based (who can vet original sources)
- Consensus (N approvals for contentious changes)
- Policy-based (auto-promotion rules based on provenance chain integrity)

### Implications for Implementation
- Every wiki page/section needs **citation metadata** — what sources it relies on
- Need a **dependency graph** between pages/claims
- Need **deprecation detection** — when a vetted source is challenged, trace dependents
- The lint operation becomes critical — it validates provenance chains

## Prior Art: PHAT-TOAD Framework (from /srv/PHAT-TOAD-with-Trails/)

The user has an existing multi-agent coordination framework that directly informs prd2wiki's design. Key patterns to carry forward:

### Steward Council Protocol
- **Phased gates** with mechanical enforcement (not advisory)
- **Cross-comprehension tests**: Agent A describes Agent B's system; B corrects; diff reveals actual understanding
- **Break tests**: "show me what breaks" as comprehension verification
- Steward's value is in **questions, not answers** — structure over intelligence
- **Independent parallel questioning** before letting agents hear each other

### Agent System Rules
- **Provenance tags**: `[UNVERIFIED — carried from agent N]` on inherited knowledge
- **Constraint declarations** are mandatory artifacts (hard constraints, anti-patterns with WHY, test invariants, performance contracts)
- **Anti-patterns** (6.1-6.8): Confident Architect, Premature Builder, Shallow Agreement, Architectural Deference, Performative Compliance, Phase Blurring, Premature GO, Clean vs Complete
- **"No concerns" is a red flag** — must walk through specifics

### Agent Handoff Protocol
- **No-mutate rule**: delivered documents are immutable, corrections go in new documents
- **Comprehension verification** before any cross-agent work
- Exchange → comprehend → verify → scope → plan → agree → reflect

### Inspector Pattern
- Analyzes agent sessions against framework rules
- Produces compliance reports with evidence-backed assertions
- Generates targeted questionnaires for agents
- Works WITH human steward, not autonomously

### What This Means for prd2wiki

The wiki's steward agents should implement these patterns:
1. **Challenge mechanism** = steward council's cross-comprehension + break tests
2. **Provenance tags** = wiki's citation/trust chain metadata
3. **No-mutate rule** = wiki's revision history (never overwrite, always new version)
4. **Constraint declarations** = wiki page metadata for hard requirements
5. **Inspector pattern** = wiki's lint operation, checking pages against consistency rules
6. **Anti-pattern detection** = steward agents watching for known failure modes in wiki contributions
7. **Mechanical gates** = authorization model for truth-branch promotion

## Architecture Decision: Sidecar/Adapter Pattern

The wiki core exposes a clean API. All external interfaces attach as **sidecars/adapters** — not compiled in:

- **MCP sidecar** — mandatory, but deployable independently. Speaks MCP to agents, REST to wiki core.
- **Web UI** — serves the human WYSIWYG interface, calls wiki core API
- **Future adapters** — CLI, webhooks, Slack, etc. — all attach the same way

This means:
- Wiki core is a single Go binary with a REST/gRPC API
- Sidecars can be written in any language
- MCP sidecar can be updated/replaced without recompiling the core
- Adapters are independently deployable, scalable, testable
- Aligns with PHAT-TOAD Rule 1: one core, adapters absorb differences

## Design Principle: Familiar, Trusted Building Blocks

> "I would like people to look at the project and not reject it as alien, but as familiar and building on blocks that are easily understood and TRUSTED. We are forging new ground..."

This is a foundational design constraint. prd2wiki should feel like a natural evolution of things people already know and trust — git, markdown, wikis, PRs, conventional commits. The novel parts (provenance chains, steward agents, challenge mechanisms) should be expressed through familiar patterns, not alien abstractions.

**Implication:** Every design choice should ask: "Does an established, trusted standard already solve this?" Only invent where nothing exists.

## Access Control Analysis — Journey to RBAC

### What We Explored (and Why We Rejected It)

1. **Page-level RBAC with identity pages** — overengineered. Conditional policies, scope syntax, identity-as-wiki-pages creating bootstrap problem. Enterprise IAM, not a wiki.

2. **Unix UGO (User/Group/Other)** — no wiki system uses it. One-group-per-page limit is fatal for multi-team. Also: git doesn't preserve file ownership, so UGO would be a reimplemented abstraction pretending to be familiar. False familiarity.

3. **UGO + POSIX ACLs** — same problem. Building a custom permission system and naming it after Unix conventions doesn't make it Unix. The implementation cost is the same regardless of naming.

### Industry Research: What Every Wiki Actually Does

Surveyed MediaWiki, Confluence, Outline, Gitea, BookStack, Notion, Wiki.js. **All converge on the same pattern:**

**RBAC with hierarchical content scoping.** Three layers:

1. **System role** — what kind of user are you (admin, editor, viewer, guest)
2. **Container scope** — what can your group do in THIS project/space
3. **Page override** — does THIS specific page restrict further (optional allowlist)

No wiki uses UGO. No wiki uses per-page ACLs as the primary model. MediaWiki's flat wiki-wide groups are widely considered a limitation.

### Decision: Standard RBAC

**4 roles** (industry sweet spot):
- `admin` — manage users, config, structure
- `editor` — read, write, propose, challenge, approve CRs
- `viewer` — read only
- `guest` — access to specifically shared pages only

**Scoped by project** (the primary permission boundary):
```yaml
projects:
  project-a:
    admins: [jane@example.com]
    editors: [team-engineering, agent-lint-3@service.internal]
    viewers: ["*"]
```

**Page-level overrides** (optional, restrict-only):
```yaml
access:
  restrict_to: [team-security, jane@example.com]
```

**Auth**: OIDC/OAuth for both humans and agents. Same permission evaluation for API and UI. Tokens are user-scoped.

**Quality gates are separate**: Branch protection (provenance validation, trust levels, required reviewers) = merge rules, not access rules.

## Page Schema & Submission Modes

Every page conforms to a schema defined per `type`. Schemas are validated by the wiki core on every write.

Three submission modes give explicit control over what the system does with input:

| Mode | Label | Behavior |
|---|---|---|
| **Verbatim** | `Submit [Do Not Mutate]` | Stored exactly as entered. No corrections. Schema validation runs but non-conforming pages are saved with `conformance: pending` flag. Original voice preserved. |
| **Conform** | `Submit [Correct & Format]` | Spelling, grammar, formatting corrected. Frontmatter completed to match schema. **No content changes** — no summarization, no rewriting, no context additions. Meaning untouched. |
| **Integrate** | `Submit [Reason & Merge]` | System actively reasons about content. May merge with existing pages, update cross-references, add provenance links, create decision blocks, resolve duplicates. Full Karpathy-style ingest. |

The conforming pipeline for `conform` and `integrate` modes:
1. Schema validation — required frontmatter fields present? Auto-populate what can be inferred.
2. Format normalization — spelling, grammar, markdown structure (conform + integrate)
3. Integration reasoning — cross-reference, identify duplicates/contradictions, propose merges (integrate only)
4. **Mandatory diff preview** — user/agent sees exactly what changed before confirming. Can accept, reject, or switch to verbatim.

Non-conforming pages are allowed. Verbatim submissions that don't match schema get stored with a flag. Lint stewards will flag them. The raw input is never lost — preserved in git history even if later conformed.

Same three modes available via MCP: `wiki_propose` tool accepts `intent: "verbatim" | "conform" | "integrate"`.

## Media & Attachments

**Storage: Git LFS** — industry standard for binary files in git. Media is versioned alongside content in the same commit history.

**Directory structure — sibling `_attachments/` pattern:**
```
pages/
  product/
    roadmap.md
    roadmap/
      _attachments/
        architecture-diagram.png
        q3-plan.pdf
  _shared/
    images/
      company-logo.png
```

**References: standard relative markdown paths.** No UUIDs, no custom syntax. Portable, works in any renderer, readable by humans and agents.

**Source code** files are attachments, never pages. Pages are documents.

**Size limits** enforced by git hooks: warn >10 MB, reject >50 MB. Large video/audio: link externally.

**File type whitelist** enforced by hook: images (png, jpg, gif, webp, svg), documents (pdf), audio/video (mp3, mp4, webm, ogg), data (csv, json, yaml, toml), code (py, js, ts, go, etc.). Reject: exe, dll, .DS_Store, node_modules.

**Versioning:** Git history IS the version history. No separate media versioning system.

**Binary diffing:** Don't try — show version list with timestamps and authors, side-by-side thumbnails for images. This is what every production wiki does.

## Pages as Build Artifacts — Source Pinning and Self-Healing

### The Core Insight

Every page is an **as-built artifact** with a manifest. Like a Docker image with a Dockerfile — you know exactly what went into it, and you can tell when the inputs have changed. The wiki doesn't just version content — it **tracks the validity of content against its sources continuously**.

This is the provenance chain made concrete. Every page records exactly what it was built from, with checksums pinning the exact versions. When any source mutates, the chain breaks and the system knows about it automatically.

### How It Works — Walkthrough

**Day 0: Page is created with pinned sources**

```
Page: PRD-042 (as-built)
  │
  ├── Built from:
  │   ├── wiki://project-a/auth-research (checksum: abc123, v3)
  │   ├── wiki://project-b/api-spec (checksum: def456, v7)
  │   ├── https://datatracker.ietf.org/doc/html/rfc6749 (checksum: 789ghi, retrieved: 2026-04-01)
  │   └── /srv/project-a/src/auth/handler.go (checksum: jkl012, commit: fa3b2c1)
  │
  └── Valid ONLY for these exact versions
```

All sources match their checksums. PRD-042 is valid, trust_level: 3.

**Day 5: A source changes**

Someone updates `project-b/api-spec` (v7 → v8). The checksum no longer matches.

```
PRD-042 status: STALE
  │
  └── Source changed: wiki://project-b/api-spec
      Expected: def456 (v7)
      Current:  xyz789 (v8)
      Impact: UNKNOWN — needs steward review
```

**What happens automatically:**
1. PRD-042 gets flagged `status: stale` with a `[!stale]` block explaining which source changed
2. Trust level drops (L3 → L1 — "was valid, no longer verified against current sources")
3. Steward agent is notified
4. All pages that cite PRD-042 are also flagged as potentially impacted (**cascading**)

**What the steward does:**
1. Diffs the old and new versions of the changed source (`api-spec` v7 vs v8)
2. Determines impact on PRD-042:
   - **No impact:** The change in api-spec doesn't affect what PRD-042 references. Steward clears the flag, updates the checksum to v8, trust restored.
   - **Partial impact:** Some claims in PRD-042 are still valid, some are not. Steward marks the specific sections, explains what's accurate and what's not.
   - **Full invalidation:** The change fundamentally breaks PRD-042's claims. Steward flags it for the owning agent/human to rewrite.

### Frontmatter Representation

```yaml
provenance:
  sources:
    - ref: "wiki://project-b/api-spec"
      version: 7
      checksum: "sha256:def456..."
      retrieved: 2026-04-01
      status: stale                    # was: valid
      stale_since: 2026-04-05
      current_version: 8
      current_checksum: "sha256:xyz789..."
      impact_assessment:
        assessed_by: steward-lint-3
        date: 2026-04-05
        verdict: partial               # none | partial | full
        details: "OAuth flow unchanged. Token format changed — section 3.2 is now inaccurate."
        sections_affected: ["3.2"]
        sections_cleared: ["1", "2", "3.1", "4"]
```

### Self-Healing

When the owning agent updates section 3.2 to reflect the new api-spec v8, the source entry updates to `status: valid` with the new checksum. Trust level restores. Downstream pages that cited PRD-042 get re-evaluated — the healing cascades just like the invalidation did.

### Self-Deprecation

If a page's sources are so thoroughly invalidated that the page can't be trusted at all, the system can auto-set `status: deprecated` with a clear explanation:

```markdown
> [!deprecated] Auto-deprecated by steward-lint-3
> **Date:** 2026-04-05
> **Reason:** 3 of 4 sources have been superseded. RFC 6749 replaced by RFC 9700.
> api-spec v7 replaced by v8 with breaking changes. auth-research retracted.
> **Action required:** Owner must rebuild this page against current sources.
```

### Source Types and How They're Tracked

| Source Type | How Pinned | How Staleness Detected |
|---|---|---|
| **Wiki page** (same project) | Page ID + version + checksum | Git hook on commit — immediate |
| **Wiki page** (cross-project) | Project + page ID + version + checksum | Periodic poll or webhook between projects |
| **External URL** | URL + checksum + retrieval date | Periodic fetch and re-checksum |
| **Source code file** | File path + commit hash + checksum | Git hook on commit to source repo |
| **Local file** (PDF, doc, etc.) | Path + checksum, local copy in `_sources/` | Local copy preserved; staleness if original path changes |

### Local Copies and Source Preservation

External sources (URLs, files from other systems) get a **local copy** stored in the wiki when possible:
- External URLs: fetched and stored in `_sources/` alongside the page
- PDFs, documents: stored in `_attachments/` or `_sources/`
- The local copy preserves what the page was built from, even if the external source disappears or changes

This means the wiki is resilient to external source deletion — the evidence that informed a decision is preserved locally, and the checksum proves it matches what was originally retrieved.

### Why This Is Novel

No existing system does this. Version control tracks what changed. Knowledge graphs track relationships. But no production system:
- Pins pages to exact source versions with checksums
- Automatically detects when sources mutate
- Cascades staleness through dependency chains
- Has steward agents that assess impact and selectively clear/invalidate
- Self-deprecates pages when sources are thoroughly invalidated
- Self-heals when owning agents update affected sections

Every page is a **verifiable build artifact**. The wiki maintains continuous validity, not just version history.

## Strict Schema as the Foundation

### The Principle

Strict schema enforcement at the boundary means the system can trust its own data. Every page on the truth branch conforms to a known schema. This makes everything downstream mechanical rather than interpretive:

- Provenance checking — fields are always there, always in the same place
- Checksums — system knows exactly where to find source references
- Staleness detection — automated, no guessing about which fields to check
- Cross-referencing — every page has `id`, `type`, `status`, `provenance.sources` predictably
- Steward agents — process structured data, not free-form text

### Schema Enforcement by Branch

| Branch | Schema Enforcement |
|---|---|
| `truth` | **Strict.** Full schema compliance required. Branch protection rule — merge blocked if validation fails. |
| `draft/*` | **Flexible.** Non-conforming pages saved with `conformance: pending`. Work in progress is allowed to be messy. |
| `challenge/*` | **Partial.** Challenge metadata must conform (evidence, references). Challenged content itself may be in flux. |
| `ingest/*` | **Flexible on input, strict on output.** Raw ingested material can be anything. The ingest steward's job is to produce schema-conforming pages from it. |

### Schema Enforcement by Submission Mode

| Mode | Schema Enforcement |
|---|---|
| **Verbatim** | Validated but not blocked. Saved with `conformance: pending`. Can exist on draft branches, cannot merge to truth. |
| **Conform** | Auto-fixed to pass validation. Missing fields populated where inferrable, formatting corrected. Must pass before save. |
| **Integrate** | Full compliance required. System ensures all provenance, references, and metadata are complete. |

### What This Enables

Because the truth branch is always fully schema-conformant:
- The computed index can be rebuilt mechanically from any truth-branch commit
- Source pinning and self-healing work reliably — checksums are always present
- Cascading invalidation is complete — no pages with missing provenance fields to skip
- Steward agents operate on structured data, not heuristic interpretation
- The wiki can make guarantees about its own truth: "every page on this branch has complete provenance"

This is the equivalent of a type system for the wiki. Draft branches are dynamically typed (flexible, fast). Truth branch is statically typed (strict, reliable).

|---|---|---|
| **Standard** | Validation errors block write. Caller must fix and retry. | `Submit [Correct & Format]` — system enforces schema |
| **Insistent** | Caller disagrees with objections. Entry saved with `contested` flag, objections attached as metadata. Enters cleanup queue. | `Submit [Do Not Mutate]` — saved as-is but flagged non-conforming |
| **Override** | Admin only. Write unconditionally, log the override. | Admin merge to truth bypassing gates (rare, audited) |

**Key insight: "Authority negotiation, not gatekeeping."** The librarian doesn't block writes — it flags contested entries. This aligns with prd2wiki's principle that draft branches are flexible while truth is strict.

### Progressive Classification (3-tier)

Pippi classifies incoming data using escalating cost:
1. **Rule-based heuristics** (< 1ms) — pattern matching against known shapes
2. **Vector similarity** (< 10ms) — embed and compare against existing entries
3. **LLM escalation** (budgeted) — only when confidence < 0.7 after tiers 1+2

**For prd2wiki:** The `conform` and `integrate` submission modes should use similar progressive classification. Schema validation is tier 1 (instant). Provenance checking and cross-referencing is tier 2. Full reasoning about integration is tier 3 (LLM, budgeted).

### Prior Art Engine

The librarian suggests metadata for new entries based on similar existing entries:
- Top-K similar entries → field frequency analysis
- If 90%+ of similar entries have a field, it's "effectively required"
- Cold-start: suppress suggestions until >= 5 entries of same type exist

**For prd2wiki:** When an agent creates a new requirement page, the wiki could suggest: "18/20 similar requirement pages have a `security_impact` field — consider adding one." This is the `conform` mode's intelligence.

### Vocabulary Management

Explicit vocabulary table with canonical terms, aliases, and fuzzy matching (> 0.85 similarity = normalize to existing term). Prevents "network-device" vs "Network Device" vs "net_device" drift.

**For prd2wiki:** Tags, relationship predicates, and document type names should have a canonical vocabulary. The conform mode normalizes incoming terms. New terms are logged for review.

### Deduplication: Keep-Both-Linked

Near-duplicates with conflicting data are linked, never auto-merged. Human review resolves conflicts. Merge execution is explicit with per-field resolution.

**For prd2wiki:** When two agents submit similar content, the system links them as potential duplicates rather than auto-merging. Steward agents or humans resolve. This is safer than silent merge.

### Fast-Track Bypass

Typed, known-shape data from registered processes bypasses the full validation pipeline. Only token verification + schema matching.

**For prd2wiki:** System-internal writes (steward lint results, computed index updates, automated staleness flags) should bypass the full conform pipeline. They're already structured and trusted.

### What prd2wiki Should Adopt From Pippi

1. **Authority model** (standard/insistent/override) maps cleanly to submission modes
2. **Progressive classification** for the conform/integrate pipeline
3. **Vocabulary management** with canonical terms and fuzzy normalization
4. **Keep-both-linked deduplication** — never auto-merge conflicting content
5. **Severity levels in validation** (error/warning/info) — errors block truth merge, warnings are surfaced, info is optional
6. **Fast-track bypass** for typed internal processes
7. **Contested metadata schema** — attach objections to entries rather than blocking them

### What prd2wiki Does Differently

- Pippi uses LanceDB (vector DB) + SQLite. prd2wiki uses git + SQLite index.
- Pippi's librarian is a runtime service with an in-memory pipeline. prd2wiki's validation runs at commit/merge time.
- Pippi's vocabulary lives in SQLite. prd2wiki's vocabulary could be wiki pages (`type: vocabulary`) — recursive architecture.
- Pippi doesn't have branching or truth/draft separation. prd2wiki's branch model is the novel contribution.

## Librarian as Core Component

### Decision

The Pippi Knowledge Librarian pattern is adopted as a core component of prd2wiki, not a sidecar. The librarian sits between the Core API and storage (git + indexes). Every write passes through it.

### Architecture with Librarian

```
Git Bare Repos (markdown files = the actual content)
       ▲
       │ read/write
       │
   Librarian (classify, validate, normalize, dedup, index)
       │
       ├──▶ SQLite index (metadata, FTS, provenance graph)
       ├──▶ LanceDB index (embeddings, page pointers)
       │
   Core API
       │
   ┌───┼───────────┐
 Web UI  MCP     Adapters
       Sidecar
```

### Read Flow (no librarian)

Browser or agent requests a page → Core API reads from git → returns markdown + metadata. The librarian is not involved in reads.

### Write Flow (librarian involved)

1. User/agent submits content with an intent (verbatim/conform/integrate)
2. Core API passes to librarian
3. Librarian runs its pipeline based on intent:
   - **Verbatim**: schema validation (flag, don't block) + index updates
   - **Conform**: validate + normalize spelling/format/tags + dedup check + return diff for preview
   - **Integrate**: full pipeline — validate + normalize + cross-reference + reason about placement + suggest merges + return diff
4. **Mandatory diff preview** — the librarian returns proposed changes, NOT committed yet
5. User/agent reviews diff and accepts, edits, or switches to verbatim
6. On acceptance: librarian commits .md to git, updates SQLite, updates LanceDB
7. Done

### Key Principle

The librarian never silently changes content. Every mutation is previewed. The user/agent is always in control. The librarian suggests, the human or agent decides.

### Vector DB Role

LanceDB stores embeddings (mathematical representations) and pointers back to git, NOT the actual markdown content. Like a library card catalog — tells you where to find the book, but is not the book. Can be deleted and rebuilt from git at any time.

Powers: semantic search, dedup detection, prior art suggestions, cross-reference discovery.

### Mapping to Pippi's Authority Model

| prd2wiki Submission Mode | Pippi Authority Level | Librarian Behavior |
|---|---|---|
| Verbatim | Insistent | Save with flags, don't block |
| Conform | Standard | Enforce schema, normalize, preview diff |
| Integrate | Standard + full pipeline | Reason, cross-reference, suggest merges |
| Admin override | Override | Write unconditionally, log |

## Hard References vs. Soft References

### The Distinction

Two fundamentally different kinds of references, served by different systems:

**Hard References** — explicit, intentional, provenance-tracked:
- Declared in frontmatter `provenance.sources` by the author
- Checksummed and version-pinned
- Part of the trust chain — staleness detection and cascading invalidation apply
- Stored in git (frontmatter YAML)
- "I built this page using these sources"

**Soft References** — discovered by the system, not declared by the author:
- Generated by the vector DB (semantic similarity)
- "Pages related to this topic that you might find useful"
- NOT part of the provenance chain — no checksums, no staleness tracking
- Computed at read time or periodically, stored in the computed index
- Discovery aid, not a trust claim

### Soft Reference Dismissal

Users and agents can mark soft references as not valid or block them — simple checkbox with optional reason:

```
Soft References (discovered):
  ☑ PRD-099 "Session Management" (0.89 similar)
  ☐ project-b/auth-design (0.84 similar)
      Reason: "Different auth paradigm, not applicable"
  ☑ decision/jwt-vs-opaque (0.78 similar)
```

Checked = valid/useful. Unchecked = dismissed. Reason is optional but helps steward agents understand why.

Dismissals stored in a sidecar file (`PRD-042.soft-refs.yaml`) alongside the page, git-tracked:

```yaml
dismissed:
  - ref: "project-b/auth-design"
    dismissed_by: "jane@example.com"
    date: 2026-04-05
    reason: "Different auth paradigm, not applicable"
accepted:
  - ref: "PRD-099"
    accepted_by: "jane@example.com"
    date: 2026-04-05
```

Key behaviors:
- Dismissed soft refs stop appearing for that page
- Accepted soft refs are pinned (persist even if similarity drifts)
- Dismissals are git-tracked — auditable, reversible
- New pages may surface new suggestions; previously dismissed refs don't resurface unless substantially rewritten

### Soft-to-Hard Promotion

An accepted soft reference can be **promoted to a hard reference** — one action adds it to frontmatter with checksum. The ref enters the provenance chain, staleness tracking begins.

Discovery → validation → trust pipeline:
```
Vector DB surfaces soft ref → user reviews → accept/dismiss → optionally promote to hard ref
```

### Reference Trees

References are not flat lists — they are navigable trees. A page cites sources, those sources cite their own sources, forming a graph. The UI renders this as an expandable tree rooted at the current page.

Default view: collapsed (direct references only). Click ▶ to expand and see deeper levels. Each node shows version, checksum status, and trust level. Stale/contested flags propagate upward visually — if a deep dependency is stale, the ⚠ indicator shows at every level so you can spot problems without fully expanding.

Soft references also expand into trees — expanding a soft ref reveals ITS hard references, showing WHY the system thinks it's related.

API supports a `depth` parameter for recursive traversal. Default depth=1, UI lazy-loads deeper levels on expand. Data sourced from SQLite provenance graph (hard refs) and vector DB (soft refs).
