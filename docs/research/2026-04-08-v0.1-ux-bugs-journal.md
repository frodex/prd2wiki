# UX Bug Discovery Journal v0.1
**Date:** 2026-04-08
**Session:** SESSION-1 (continued)
**Status:** In Progress
**Preceding:** none
**Changes from v0.0:** Initial journal from first real-user testing session

## Context

First real-user testing of prd2wiki web UI. User created pages via browser and via MCP. Multiple bugs discovered that would have been caught by integration testing with actual browser workflows.

## Bug 1: JSON Payload Structure Mismatch

**Discovered:** User submitted a page with title "Hello World Test 02a" — got "Title is required" error.

**Root cause:** The JavaScript `collectFormData()` wrapped fields in a `frontmatter` object:
```json
{"frontmatter": {"title": "Hello World", "id": "...", ...}, "body": "..."}
```
But the Go API's `CreatePageRequest` expected flat top-level fields:
```json
{"title": "Hello World", "id": "...", "body": "..."}
```

The title was silently lost in nesting. The server saw an empty title.

**Fix:** Flattened the JS payload to match the API struct.

**Lesson:** Frontend and backend were built by different subagents with no integration test that actually submitted a form and verified the round-trip. Need end-to-end tests that POST JSON matching what the browser sends.

## Bug 2: Empty ID Creates Broken Page

**Discovered:** User left the ID field blank expecting auto-generation. The system created a file named `pages/.md` (empty ID) which couldn't be viewed, edited, or deleted through the UI.

**Root cause:** No ID auto-generation. The librarian accepted empty ID in verbatim mode (by design — verbatim doesn't block on schema errors), but the resulting file path was broken.

**Fix:** Added `generateID()` function in the librarian that slugifies the title. Falls back to random `page-{date}-{hex}` if no title. Runs before any submission mode.

**Lesson:** Schema validation errors in verbatim mode should still prevent structurally broken outcomes (empty filenames). Validation vs. "don't block" needs a middle ground: some constraints are structural (valid filename), not content quality (missing provenance).

## Bug 3: Index Rebuild Wipes Across Branches

**Discovered:** PRD-001 (created via MCP on `draft/agent` branch) disappeared from the page list after restart.

**Root cause:** `RebuildFromRepo()` deleted ALL pages for the project before re-indexing a single branch. When main.go called it for `draft/agent` then `draft/incoming`, the second call wiped what the first indexed.

```
Startup:
  RebuildFromRepo("default", repo, "draft/agent")    → indexes PRD-001
  RebuildFromRepo("default", repo, "draft/incoming")  → DELETE FROM pages WHERE project='default'
                                                        → PRD-001 gone
                                                        → indexes only draft/incoming pages
```

**Fix:** Changed `RebuildFromRepo` to clear only entries for the specific `project + branch` being rebuilt, not the whole project.

**Lesson:** This is a classic "works in tests, fails in production" bug. The test used a single branch. Multi-branch behavior was never tested. Need a test that indexes pages on two different branches and verifies both survive.

## Bug 4: Branch Fallback Only Tried Two Branches

**Discovered:** Pages created on `draft/agent` (via MCP) returned 404 in the web UI.

**Root cause:** The web UI's `viewPage` and `editPage` handlers only tried `truth` and `draft/incoming` branches. Pages on other branches (like `draft/agent` from MCP) were invisible.

**Fix:** Extracted `readPageAnyBranch()` helper that tries `truth`, then all branches from `repo.ListBranches()`. Used by both view and edit handlers.

**Lesson:** The branch the page lives on should not matter to the reader. The UI should always find the page regardless of which branch holds it.

## Bug 5: Milkdown CDN Load Failure

**Discovered:** The editor showed a raw textarea instead of WYSIWYG.

**Root cause:** Milkdown loaded via ESM CDN imports (`esm.sh/@milkdown/kit@7/...`) failed silently in the browser. Complex ProseMirror-based editors have deep dependency trees that CDN resolution can't reliably handle.

**Fix:** Set up proper frontend build (Vite + npm) that bundles Milkdown into a single IIFE JS file. Textarea remains as visible fallback.

**Lesson:** Never rely on CDN for complex editor frameworks. Always bundle.

## Tests Needed

Based on these bugs, the following tests should be added to the build/CI:

1. **Integration test: form submission round-trip** — POST JSON matching browser format, verify page is created with correct fields
2. **Test: empty ID auto-generation** — submit with no ID, verify a valid ID is generated
3. **Test: multi-branch index rebuild** — index pages on two branches, rebuild both, verify both survive
4. **Test: readPageAnyBranch finds pages on any branch** — create page on draft/agent, verify readPageAnyBranch finds it
5. **Test: structurally invalid verbatim submissions** — empty ID in verbatim mode should still generate a valid filename
